1. Abstract

MeetUP Tool is an application designed for students so they can meet in certain places and study together

2. Introduction

Most of the students function better while studying with another one, eventually who has the same problems and preocupations as him. In more simple words, depression caused by exams can unite us.

3. Goals

  • giving users the possibility of meeting with friends and studying same things at the same time
  • ability to chat real time locally, inside application
  • ability to send notifications regarding your schedule
  • possibility to schedule events in an inside calendar

4. Web components

Web components are a set of web platform APIs that allow you to create new custom, reusable, encapsulated HTML tags to use in web pages and web apps. Custom components and widgets build on the Web Component standards, will work across modern browsers, and can be used with any JavaScript library or framework that works with HTML.
Web components are based on existing web standards. Features to support web components are currently being added to the HTML and DOM specs, letting web developers easily extend HTML with new elements with encapsulated styling and custom behavior.

Web components are based on four main specifications:

Custom Elements
The Custom Elements specification lays the foundation for designing and using new types of DOM elements.

Shadow DOM
The shadow DOM specification defines how to use encapsulated style and markup in web components.

HTML Template
The HTML template element specification defines how to declare fragments of markup that go unused at page load, but can be instantiated later on at runtime.

8. Actors and use cases

  • Actors:
  • 1. Application
  • 2. Calendar
  • 3. Local Storage
  • 4. Owner
  • 5. User
  • Use cases:

  1. Check in:

    1. User acceses his account.
    2. User is welcomed by a welcome page and he enters the location and field of study.
    3. The application sets in local storage the options chosen by the user.
    4. The page will redirect the user to home page, where he can see the feed of his friends

  2. Edit calendar:

    1. User accesses the application.
    2. User clicks on the Calendar page
    3. User has to select a date, manually or jumping straight to it.
    4. User can create an event
    5. The application stores in local database the event crated

  3. Calendar jump:

    1. User accesses the application.
    2. User selects the year and month to view
    3. User presses the button jump to
    4. The calendar will be displayed how the user selected

  4. Add event:

    1. User accesses the Calendar Page.
    2. User clicks on the "Add event" button.
    3. User has to select his options for the event.
    4. The application adds the event in local storage.
    5. The application displays the event on the page.

  5. Remove event:

    1. User accesses the application.
    2. User clicks on the "Remove event" button.
    3. The application removes the event from local storage.
    4. The application deletes the event from the page.

  6. Search Location:

    1. User access My Profile page
    2. User types in the search bar the name of the location that he is looking for
    3. User can see how many of his friends are at that location at the moment
    4. User goes and meet his friends

  7. Real time chat

    1. User searches for friends through the search bar
    2. User hits the Message button
    3. Application creates a small chat head in the right bottom corner
    4. User starts chatting in real time with his friend

9.Storage solution

Aproximating the fact that the application will not need a lot of amount of memory, we chose to store data in local storage.

10. Resource representation

The singleton design pattern is used when it comes to Calendar class. The calendar contains data about the events and the options, which are also part of the model of the application. There is also an EventManager class which handles the event opperations.
